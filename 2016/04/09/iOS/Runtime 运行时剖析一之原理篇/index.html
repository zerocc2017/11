Runtime 运行时剖析一之原理篇

Posted on 2016-04-09 | Post modified: 2018-01-15 | In iOS - 系统汇总

运行时 Objective-C 语言设计架构的核心之一。运行时系统（runtime 库）是由 C 语言和汇编语言（消息传递机制由汇编写成)编写封装实现的，它赋予了 Objective-C 语言动态特性的特征；它使得 Objective-C 语言具有了面向对象的能力。

一、Objective-C 动态语言

1.1、动态语言 & 静态语言

动态语言、静态语言是指动态类型语言和静态类型语言。

  1. 动态类型语言：是指在运行期间才去做数据类型检查的语言，程序在运行时可以改变其结构，新的函数可以被引进，已有的函数可以被删除等在结构上的变化。在用动态类型的语言编程时，不用给任何变量指定数据类型，该语言会在你第一次赋值给变量时，在内部将数据类型记录下来。Python 和 Ruby 就是一种典型的动态类型语言，但是 OC 还是要声明数据类型等等呀 (有待考证编译器和解释器等方面官方文档疑问一)。

  2. 静态类型语言：它与动态类型语言刚好相反，它的数据类型是在编译其间检查的，也就是说在写程序时要声明所有变量的数据类型，函数同样一定是需要实现而不只是有声明就可以。C、C++、JAVA、C# 都是静态类型语言。

1.2、OC 动态特性

Objective-C 作为动态语言，它将很多静态语言在编译和链接时期做的事放到运行时来处理。静态语言函数一般在编译时就已经把参数信息和函数实现打包到编译后的源码中了；而 OC 的函数属于动态调用过程，运行时机制使其只有在真正运行的时候才根据函数的名称进行调用，在编译阶段，OC可以调用任何函数，即使这个函数并未实现，只要声明过就不会报错。

OC 中函数的调用使用的是消息机制(发送消息，方法调用)。调用一个实例的方法，所做的是向该实例的指针发送消息，实例在收到消息后，从自身的实现中寻找响应这条消息的方法。

OC 的这种动态性主要体现在三方面：动态类型、动态绑定、动态加载。

  1. 动态类型：简单点说就是 id 类型。动态类型是跟静态类型相对的，像内置的明确的基本类型都属于静态类型(int、NSString等)。静态类型在编译的时候就能被识别出来；而动态类型就编译器编译的时候是不能被识别的，要等到运行时(run time)，即程序运行的时候才会根据语境来识别。

  2. 动态绑定：主要在函数调用方面，像静态语言一般在编译的时候就已经将将要调用的函数的函数签名都告诉编译器了，运行时不能改变。OC 中没有函数的概念，而是方法的调用（消息机制），编译阶段并不能决定什么，只有到运行的时候才动态地添加函数调用，调用什么方法，需要传什么参数进去，这就是动态绑定。方法的实现，运行阶段必须用 SEL 变量绑定一个方法，而这个 SEL 变量就代表一个方法的引用，OC 中的 SEL 并不是 C 语言里面的函数指针，它只是一个整数，是该方法的 ID 编号，@selector() 就是取这个方法的方法编号去查找匹配响应方法的消息，而不是根据函数名的字符串去匹配查找的；所以动态绑定的特点不仅方便，而且高效。

  3. 动态加载：是指根据不同设备加载所需要的资源，体现在兼容性方面，根据不同的机型做适配。例如不同设备上加载原图片、@2x的图片。

二、Runtime 运行时简介

runtime 简称运行时。其是系统运行的时候的一些机制，本文主要分析 Objective-C 中类与对象的本质和方法如何实现（消息传递过程）。

Objective-C 是基于C语言加入了面向对象特性 (OC是重度面向对象语言) 的动态语言，它是如何具有面向对象的能力的呢？对象的方法又是如何动态调用的呢？OC中对象是通过 C 语言的结构体来表示，对象的方法实现通过 C 函数来实现；消息传递机制由汇编写成，以上 OC 特性决定了其不仅需要一个编译器(疑问思考其编译的过程)，而且还需要运行时系统（运行时库）来动态创建类和对象和进行消息传递。

三、Objective-C 中类与对象

3.1、类与对象的概念

  • 类是对同一类事物高度的抽象，类中定义了这一类对象所应具有的静态属性（属性）和动态属性（方法）。
  • 类是抽象的概念，我们一般是自定义一个类，只是定义其不占用内存，而对象是具体的，我们创建一个对象，它是实实在在存在的并且占有内存空间。
  • 对象就是该类的一个实例，对象的创建就是类的实例化过程。

3.2、Objective-C 中的类

在 Objective-C 类继承体系中，有 NSObject 这么一个根类；它是所有类的父类(NSProxy 类抛开不考虑)，其它类都从 NSObject 继承一套基本的接口到 Objective-C 运行时体系中。

3.3、解剖类的结构体 objc_class

3.3.1、Class isa 类结构体中的第一个元素（元类）

从上面分析知道一个类(class)实际上是 objc_class 结构体,但是这个结构体中又包含了同样的 isa 。说明了在 Objective-C 中，类本身就是一个对象(万物皆对象)，类对象的类我们称之为 metaClass。

metaClass 也是 class，其本质也是 objc_class 结构体。它是类对象的类。

  1. metaClass的作用是什么？

    runtime 的文档中对 metaClass 相关解释如下：

    • 实例方法和实例变量应该被加到类中。类方法是被加到元类中。
    • 元类不支持添加实例变量。
    • 如果要获取类方法，是通过元类来获取的。

    metaClass 的作用就是保存类方法。class 是实例对象的类，所有的实例变量、实例方法都保存在 class 中，而 metaClass 是类的类，所以所有的类变量（OC是没有类变量，如果有应该是这样的逻辑）、类方法都保存在 metaClass 中。

    在 Python 和 Ruby 语言中也有 metaClass 的概念。要更好理解这个概念可看看 metaClass wiki。

3.3.4、 objc_cache 方法缓存

类的结构体中 cache 为方法缓存，其提高了方法调用性能。每当实例对象接收到一个消息时,它不会直接在 isa 指向的类的方法列表中遍历查找能够响应消息的方法，因为这样效率太低了，而是优先在 cache 中查找。Runtime系统会把被调用的方法存到 Cache 中（理论上讲一个方法如果被调用，那么它有可能今后还会被调用）method_name 作为 key，method_imp 作为 value 给存起来，下次查找的时候效率更高。这和计算机组成原理中学过的 CPU 绕过主存先访问 Cache 的道理类似吧(我是这么理解的)，高速缓存(cache) ->内存 -> 虚拟内存 -> 磁盘。

4.2、消息流程(方法实现内部过程)

调用方法的方式有两种：

  • 第一种的方式调用方法，如果一个对象无法按上述正常流程接受某一消息时，就会启动所谓 “消息转发(message forwarding)” 机制，通过这一机制，我们可以告诉对象如何处理未知的消息。默认情况下，对象接收到未知的消息，会导致程序崩溃，通过控制台，我们可以看到以下异常信息：这段异常信息实际上是由NSObject的doesNotRecognizeSelector方法抛出的。不过，我们可以采取一些措施，让我们的程序执行特定的逻辑，而避免程序的崩溃。

  • 第二种方式调用方法，则需要等到运行时才能确定object是否能接收message消息。如果不能，则程序崩溃。通常，当我们不能确定一个对象是否能接收某个消息时，会先调用respondsToSelector:来判断一下。如下代码所示：

  1. 检测这个selector是不是要忽略。比如 Mac OS X 开发，有了垃圾回收就不理会retain, release这些函数了。

  2. 检测这个target是不是nil对象。ObjC的特性是允许对一个nil对象执行任何一个方法不会Crash，因为会被忽略掉。

  3. 上面检测都通过则开始查找这个类的IMP,先从cache里面找,完了找得到就跳到对应的函数去执行。

  4. 如果cache找不到，通过对象的isa指针获取到类的结构体，然后在方法分发表里面查找方法的selector (方法分发表既是：class中的方法列表method_list，它将方法选择器和方法实现联系起来)。

  5. 如果分发表找不到，objc_msgSend 结构体中指向父类的指针找到其父类，并在父类的分发表去找方法的selector，会一直沿着类的继承体系到达NSObject类。一旦定位到selector，函数会就获取到了实现的入口点，并传入相应的参数来执行方法的具体实现,并将该方法添加进入缓存中。如果最后也没有定位到selector，则会走消息转发流程。

  6. 在异常抛出前，消息转发 Objective-C 的运行时会给你三次拯救程序的机会：动态方法解析、重定向接收者、完整的消息转发。

4.3、消息转发

4.3.1、动态方法解析

对象在接收到未知的消息时，首先会调用所属类的类方法 +resolveInstanceMethod:(实例方法)或者 +resolveClassMethod:(类方法)。在这个方法中，我们有机会为该未知消息新增一个“处理方法”(或者说函数实现)，通过运行时class_addMethod函数动态添加到类里面就可以了。

  • + (BOOL)resolveInstanceMethod:(SEL)sel; 解析实例方法
  • + (BOOL)resolveClassMethod:(SEL)sel; 解析类方法
  • 通过 class_addMethod(...) 的方式将缺少的selector动态创建出来，前提是有提前实现好的IMP（method_types一致)
  • 如果 resolveInstanceMethod: 方法返回NO，运行时就会进行下一步：消息转发（Message Forwarding）。

4.3.2、重定向接收者

如果上一步动态方法解析没有处理，runtime会调用以下方法

  • -(id)forwardingTargetForSelector:(SEL)aSelector;
  • 如果该方法返回非nil的对象，则使用该对象作为新的消息接收者
    不能返回self，会出现无限循环
  • 如果不知道该返回什么，应该使用[super forwardingTargetForSelector:aSelector];
  • 这种方法属于单纯的转发，无法对消息的参数和返回值进行处理
  • 这一步合适于我们只想将消息转发到另一个能处理该消息的对象上。但这一步无法对消息进行处理，如操作消息的参数和返回值。

完整的消息转发

  • 消息转发的第一步重写方法：- (NSMethodSignature *)methodSignatureForSelector:(SEL)aSelector;为未有实现的方法的selector提供合适的方法签名然后返回这个aSelector的方法签名。
  • 当对象发送一个unrecognized 的消息时，会使用从上面方法获取的方法签名等方法信息创建一个表示消息的 NSInvocation 对象，把与尚未处理的消息有关的全部细节都封装在anInvocation中，包括selector，目标(target)和参数。
  • 然后调用- (void)forwardInvocation:(NSInvocation *)anInvocation;方法，在方法中又通过- (void)invokeWithTarget:(id)target;方法选择将消息转发给指定对象。其中 forwardInvocation: 方法的实现有两个任务：
    • 定位可以响应封装在anInvocation中的消息的对象。这个对象不需要能处理所有未知消息。
    • 使用anInvocation作为参数，将消息发送到选中的对象。anInvocation将会保留调用结果，运行时系统会提取这一结果并将其发送 到消息的原始发送者。
