iOS 开发中遇到的 Bug

Posted on 2015-07-01 | Post modified: 2018-12-10 | In iOS - 开发日常

ermination Reason: Namespace SPRINGBOARD, Code 0x8badf00d

异常代码 0x8badf00d 指示应用程序已终止的 iOS 因为看门狗超时发生。应用程序时间太长，启动、 终止，或对系统事件作出响应。一个常见的原因做在主线程上的同步联网。无论操作是线程 0 上： 需要搬到后台线程，或处理方式不同，所以，它不会阻止在主线程。

原因：程序启动时候不能超过5－6秒；

解决：优化应用启动速度，UIApplicationDelegate中的不要有好时操作，或者把耗时操作异步处理。

[{ device: ‘bb58454cee9ad7515bb779d7f1a6bc3a07e51fea241796b376**‘, status: ‘400’, response: { reason: ‘BadDeviceToken’ } } ]。服务器生产环境推送失败。

原因： apns 服务器与 app 类型环境不匹配（获取的手机的 token 环境），苹果服务器返回 BadDeviceToken；

证书与 apns 服务器类型不匹配，苹果服务器则返回 BadCertificateEnvironment 错误；

apns有两类服务器：

Development server: api.development.push.apple.com:443；

Production server: api.push.apple.com:443；

解决：生产环境推送服务器连接的 apns 苹果服务器却是开发环境的，改正后推送正常。

dyld: Library not loaded: @rpath/QuickUnifyPlatform.framework/QuickUnifyPlatform
Referenced from: /var/containers/Bundle/Application/6262F2F0-2EF3-4DF2-AF42-B065352E1C49/xxx.app/xxx
Reason: image not found
Message from debugger: Terminated due to signal 6

原因：导入第三方库，运行程序报错crash，意思是未能加载这个 .framework,镜像没有；

解决：

  1. TARGET -> General -> Embedded Binaries -> 点击 ➕ 按钮添加这个 .framework；
  2. Target -> Build Settings -> Runpath Search Paths：检查是否有 @executable_path/Frameworks 字段（xcode 默认是有的），无的话添加上；
  3. 如果还是不能解决参考 Library not loaded stackoverflow

_dispatch_once[inlined]、
0x1008a07a0 <+288>: lbr x0, [sp #64]

原因：使用单例造成的死锁，多个单例间的初始化调用中直接或间接存在相互调用。A->B->A 或者 A->B->C->A。

解决：查看代码呗，将滥用单例的类抽出打破这种循环引用。参考滥用单例之dispatch_once死锁

Implicit declaration of function ‘XXX’ is invalid in C99

原因：C 语言的 C99 编译错误，一般是出现在 C 调用苹果 API 底层接口报错。

解决：首先设置 C 语言的编译选项， TARGET -> Build Setting -> 搜索 C Language Dialect - 设置为 GNU99 [-std=gnu99],这也是 Xcode 默认的设置。其次是导入 API 的头文件。
